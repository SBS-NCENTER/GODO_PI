# 임베디드 시스템 신뢰성 점검 가이드

> Arduino, ESP8266/ESP32, STM32, Raspberry Pi 등 주요 플랫폼별 치명적 결함 패턴과 장기 운용을 위한 사전 점검 체크리스트

---

## 1. 공통 치명적 결함 패턴

임베디드 시스템 전반에서 플랫폼에 관계없이 반복적으로 발생하는 결함들이다.

### 1.1 메모리 관련 결함

| 결함 유형             | 증상                         | 원인                    | 대응                                      |
| --------------------- | ---------------------------- | ----------------------- | ----------------------------------------- |
| **스택 오버플로우**   | 랜덤 크래시, 변수 값 손상    | 깊은 재귀, 큰 로컬 배열 | 스택 사용량 모니터링, 로컬 버퍼 크기 제한 |
| **힙 프래그멘테이션** | 장시간 운용 후 `malloc` 실패 | 반복적 동적 할당/해제   | 정적 할당 사용, 메모리 풀 패턴 적용       |
| **스택-힙 충돌**      | 예측 불가 동작, 하드 폴트    | 과도한 스택+힙 사용     | 링크 스크립트에서 guard 영역 설정         |
| **버퍼 오버런**       | 인접 변수 손상, 보안 취약점  | 경계 검사 미수행        | `strncpy` 사용, 배열 인덱스 검증          |
| **댕글링 포인터**     | 간헐적 크래시                | 해제된 메모리 참조      | 해제 후 NULL 대입, 소유권 명확화          |

**실전 팁 — 메모리 여유 확인 코드 (Arduino)**

```cpp
// 런타임 남은 SRAM 확인
int freeMemory() {
    extern int __heap_start, *__brkval;
    int v;
    return (int)&v - (__brkval == 0 ? (int)&__heap_start : (int)__brkval);
}

void loop() {
    if (freeMemory() < 200) {
        // 경고: 메모리 부족 임박
    }
}
```

### 1.2 워치독 타이머(WDT) 미적용

장기 운용 시스템에서 **워치독을 적용하지 않는 것은 가장 흔한 설계 결함** 중 하나다.

- **증상**: 펌웨어가 무한 루프에 빠지거나 하드 폴트 발생 시 시스템이 영구 정지
- **대응**: 메인 루프에서 주기적으로 WDT를 리셋하고, 비정상 상태 시 의도적으로 WDT 리셋을 건너뛰어 자동 재부팅 유도

```cpp
// Arduino 예시
#include <avr/wdt.h>

void setup() {
    wdt_enable(WDTO_8S);  // 8초 워치독
}

void loop() {
    wdt_reset();  // 정상 동작 시에만 리셋
    // 메인 로직
}
```

```cpp
// ESP8266/ESP32 예시
void loop() {
    ESP.wdtFeed();  // 소프트웨어 WDT 리셋
    yield();        // 시스템 태스크 양보 (필수)
}
```

### 1.3 전원 관련 결함

| 결함 유형            | 증상                     | 원인                          | 대응                                  |
| -------------------- | ------------------------ | ----------------------------- | ------------------------------------- |
| **브라운아웃**       | 간헐적 리셋, 데이터 손상 | 모터/릴레이 구동 시 전압 강하 | 디커플링 캐패시터, 별도 전원 분리     |
| **전원 노이즈**      | ADC 값 불안정, 통신 에러 | 스위칭 레귤레이터 리플        | LC 필터, LDO 사용, 페라이트 비드      |
| **역전압 보호 미비** | IC 파손                  | 전원 역극성 연결              | 직렬 다이오드 또는 P-MOSFET 보호 회로 |
| **서지/정전기(ESD)** | 즉시 또는 지연 파손      | 외부 커넥터 통한 서지 유입    | TVS 다이오드, ESD 보호 IC             |
| **인러시 전류**      | 퓨즈 끊김, 전원 불안정   | 대용량 캐패시터 초기 충전     | 소프트 스타트 회로, NTC 써미스터      |

### 1.4 통신 관련 결함

| 통신 방식    | 주요 결함                             | 대응                                            |
| ------------ | ------------------------------------- | ----------------------------------------------- |
| **I2C**      | 버스 락업 (SDA가 LOW로 고정)          | 클럭 스트레칭 타임아웃, 버스 리커버리 루틴 구현 |
| **SPI**      | CS 핀 공유 충돌, 인터럽트 내 SPI 접근 | CS 핀 독립 관리, 인터럽트 내 SPI 사용 금지      |
| **UART**     | 프레이밍 에러, 버퍼 오버런            | 체크섬/CRC 적용, HW 플로우 컨트롤               |
| **WiFi/BLE** | 연결 끊김 후 미복구                   | 연결 상태 감시 + 자동 재연결 로직               |
| **UDP**      | 패킷 유실, 순서 보장 없음             | 시퀀스 번호, 재전송 로직, 하트비트              |

### 1.5 인터럽트 관련 결함

- **경쟁 조건(Race Condition)**: ISR과 메인 루프가 같은 변수를 접근할 때 `volatile` 선언 + `cli()`/`sei()` 또는 `noInterrupts()`/`interrupts()`로 보호
- **ISR 내 장시간 처리**: ISR에서 `Serial.print()`, `delay()`, 동적 할당 금지. 플래그만 설정하고 메인 루프에서 처리
- **인터럽트 우선순위 역전**: RTOS 사용 시 우선순위 상속 뮤텍스 활용

## 2. 플랫폼별 고유 결함 패턴

### 2.1 Arduino (AVR: ATmega328P, ATmega2560)

| 결함                                | 설명                                                              | 대응                                                                 |
| ----------------------------------- | ----------------------------------------------------------------- | -------------------------------------------------------------------- |
| **String 클래스 힙 프래그멘테이션** | `String` 객체 반복 생성/파괴 시 힙 파편화 → 장시간 운용 후 크래시 | `char[]` 고정 버퍼 + `snprintf()` 사용                               |
| **EEPROM 수명 초과**                | 100,000회 쓰기 제한 초과 시 데이터 손실                           | 쓰기 횟수 최소화, 웨어 레벨링, 변경 시에만 쓰기                      |
| **analogRead() 오염**               | ADC 멀티플렉서 채널 전환 시 이전 채널 전압 잔류                   | 채널 전환 후 첫 번째 읽기 값 폐기                                    |
| **millis() 오버플로우**             | 약 49.7일 후 0으로 롤오버 → 시간 비교 로직 오류                   | `(millis() - previousMillis >= interval)` 패턴 사용 (부호 없는 뺄셈) |
| **PWM/타이머 충돌**                 | `Servo`, `tone()`, `analogWrite()`가 같은 타이머 공유             | 타이머 할당표 확인, 충돌 시 대체 라이브러리 사용                     |

### 2.2 ESP8266

| 결함                                  | 설명                                                                                | 대응                                             |
| ------------------------------------- | ----------------------------------------------------------------------------------- | ------------------------------------------------ |
| **yield() 미호출로 WDT 리셋**         | WiFi 스택이 백그라운드에서 동작하므로, 긴 루프에서 `yield()` 없으면 소프트 WDT 리셋 | 100ms 이상 블로킹 시 `yield()` 삽입              |
| **WiFi 불안정 (Auto-reconnect 실패)** | AP 연결 끊김 후 자동 복구 실패                                                      | `WiFi.setAutoReconnect(true)` + 수동 재연결 로직 |
| **Flash 쓰기 중 크래시**              | SPIFFS/LittleFS 쓰기 중 전원 차단 시 파일시스템 손상                                | 저널링 방식 쓰기, 쓰기 완료 플래그 사용          |
| **GPIO16 특수성**                     | GPIO16은 `digitalWrite` 시 다른 핀과 다르게 동작 (내부 풀업 없음)                   | 딥슬립 웨이크업 용도 외 사용 주의                |
| **ADC 단일 채널**                     | ESP8266 ADC는 1채널(A0)만 존재, 0~1V 범위                                           | 전압 분배기 필요, 멀티채널 시 외부 MUX 사용      |

### 2.3 ESP32

| 결함                               | 설명                                               | 대응                                                      |
| ---------------------------------- | -------------------------------------------------- | --------------------------------------------------------- |
| **듀얼 코어 경쟁 조건**            | Core0(WiFi)와 Core1(사용자 코드) 간 공유 변수 충돌 | `portMUX`, FreeRTOS 뮤텍스/세마포어 사용                  |
| **브라운아웃 감지 오동작**         | USB 전원 공급 시 불필요한 브라운아웃 리셋          | 브라운아웃 감지 레벨 조정 또는 비활성화                   |
| **WiFi+BLE 동시 사용 메모리 부족** | 두 스택 동시 운용 시 힙 부족                       | 하나만 사용하거나, PSRAM 장착 모듈 선택                   |
| **GPIO 스트래핑 핀 충돌**          | GPIO0, 2, 5, 12, 15는 부팅 모드 결정에 관여        | 이 핀에 외부 풀업/다운 회로 추가 시 부팅 실패 가능성 확인 |
| **Task 스택 오버플로우**           | FreeRTOS Task 생성 시 스택 크기 과소 할당          | `uxTaskGetStackHighWaterMark()`로 모니터링                |

### 2.4 STM32

| 결함                        | 설명                                                              | 대응                                                       |
| --------------------------- | ----------------------------------------------------------------- | ---------------------------------------------------------- |
| **클럭 설정 오류**          | PLL, 프리스케일러 잘못 설정 시 UART Baud rate 어긋남, 타이밍 오류 | CubeMX 클럭 트리 시각적 확인, 오실로스코프로 실측          |
| **DMA 캐시 일관성**         | Cortex-M7(STM32H7 등)에서 DCache와 DMA 버퍼 불일치                | DMA 버퍼를 non-cacheable 영역에 배치, 또는 캐시 invalidate |
| **플래시 쓰기 중 인터럽트** | 내부 플래시 프로그래밍 중 인터럽트 진입 시 하드 폴트              | 플래시 쓰기 전 인터럽트 비활성화                           |
| **HardFault 디버깅 어려움** | HardFault 진입 시 원인 파악 곤란                                  | HardFault 핸들러에서 스택 프레임 덤프 구현                 |
| **전원 도메인 미초기화**    | 백업 도메인, 아날로그 블록 전원 별도 활성화 필요                  | HAL_PWR, RCC 설정 순서 확인                                |

### 2.5 Raspberry Pi (Linux 기반)

| 결함                   | 설명                                                          | 대응                                                   |
| ---------------------- | ------------------------------------------------------------- | ------------------------------------------------------ |
| **SD 카드 손상**       | 갑작스러운 전원 차단 시 파일시스템 손상                       | Read-only FS (OverlayFS), UPS 모듈, `sync` 주기적 호출 |
| **GPIO 실시간성 부재** | Linux 스케줄러로 인해 마이크로초 단위 타이밍 보장 불가        | 실시간 제어는 별도 MCU에 위임, pigpio 라이브러리 사용  |
| **메모리 스왑 폭주**   | 물리 메모리 초과 시 스왑 발생 → SD 카드 수명 단축 + 성능 저하 | 스왑 비활성화 또는 최소화, zram 사용                   |
| **커널 패닉**          | 드라이버 충돌, OOM 킬러 동작                                  | 커널 로그 모니터링, 하드웨어 워치독 활성화             |
| **열 스로틀링**        | 과열 시 CPU 클럭 자동 하향 → 성능 급감                        | 방열판/팬 필수, `vcgencmd measure_temp` 모니터링       |
| **시간 동기화 실패**   | RTC 미장착 시 부팅 때마다 시간 초기화                         | RTC 모듈 추가 또는 NTP 동기화                          |

## 3. 장기 운용 전 점검 체크리스트

### 3.1 전원 안정성 점검

- [ ] 최대 부하 시 전원 전압 측정 (오실로스코프 권장)
- [ ] 디커플링 캐패시터 배치 확인 (IC 전원 핀 근처 100nF + 전원 입구 10~100μF)
- [ ] 전압 레귤레이터 발열 확인 (열화상 카메라 또는 접촉식 온도계)
- [ ] 전원 라인 리플/노이즈 측정 (스위칭 레귤레이터 사용 시)
- [ ] 역전압 및 ESD 보호 회로 확인
- [ ] 정전/순간 정전 시나리오 테스트 (전원 on/off 반복 100회 이상)

### 3.2 메모리 안정성 점검

- [ ] 정적 분석 도구로 버퍼 오버런, 미초기화 변수 점검
- [ ] 런타임 스택/힙 사용량 최대치 측정 (최악 조건에서)
- [ ] `String`, `new`, `malloc` 사용 지점 전수 검토 (가능하면 정적 할당으로 대체)
- [ ] EEPROM/Flash 쓰기 빈도 계산 (수명 내 쓰기 횟수 초과 여부)
- [ ] 메모리 누수 테스트 (24시간+ 연속 가동하며 여유 메모리 추이 확인)

### 3.3 통신 안정성 점검

- [ ] 통신 에러율 측정 (1만 패킷 이상 전송하며 에러/유실률 통계)
- [ ] 타임아웃 및 재전송 로직 구현 확인
- [ ] 통신 상대방 비정상 종료 시 복구 테스트
- [ ] WiFi: 신호 약한 환경(RSSI -70dBm 이하)에서 동작 테스트
- [ ] I2C: 버스 리커버리 루틴 구현 여부
- [ ] 장시간 연결 유지 테스트 (WiFi/BLE 24시간+ 연결 안정성)

### 3.4 열/환경 점검

- [ ] 동작 온도 범위 내 테스트 (최저/최고 예상 온도)
- [ ] 방열 설계 검증 (연속 최대 부하 1시간+ 시 온도 안정화 확인)
- [ ] 결로/습도 대비 (필요 시 코팅, 실링)
- [ ] 진동/충격 테스트 (이동 환경이라면)
- [ ] 커넥터/케이블 접촉 불량 시뮬레이션

### 3.5 소프트웨어 견고성 점검

- [ ] 워치독 타이머 설정 및 동작 확인
- [ ] 모든 외부 입력에 대한 범위 검증 (센서 값 이상치, 통신 수신 데이터)
- [ ] 에러 핸들링 경로 테스트 (센서 미응답, 통신 실패, 파일 쓰기 실패 등)
- [ ] `millis()` 오버플로우 대비 확인 (49.7일 기준)
- [ ] 인터럽트 내 처리 최소화 확인 (ISR 실행 시간 측정)
- [ ] 전역/공유 변수 보호 (`volatile`, 크리티컬 섹션)
- [ ] OTA(Over-the-Air) 업데이트 실패 시 롤백 메커니즘

### 3.6 장기 운용 스트레스 테스트

| 테스트 항목            | 최소 권장 시간          | 확인 사항                         |
| ---------------------- | ----------------------- | --------------------------------- |
| **연속 가동 테스트**   | 72시간 (이상적으로 2주) | 메모리 누수, 성능 저하, 자원 고갈 |
| **전원 on/off 사이클** | 1,000회 이상            | 부팅 안정성, 데이터 무결성        |
| **통신 부하 테스트**   | 24시간                  | 패킷 유실률, 연결 복구            |
| **온도 사이클**        | 50사이클 (최저↔최고)    | 납땜 크랙, 열팽창 문제            |
| **에지 케이스 주입**   | -                       | 센서 단선, 이상 전압, 동시 이벤트 |

### 3.7 생산/배포 전 최종 점검

- [ ] 펌웨어 버전 관리 체계 확립
- [ ] 부트로더 보호 (의도치 않은 덮어쓰기 방지)
- [ ] 디버그 출력(`Serial.print`) 제거 또는 비활성화
- [ ] 로그 시스템 구현 (에러 발생 시 원인 추적용)
- [ ] 필드 업데이트 방안 확보 (OTA 또는 물리적 접근)
- [ ] 복구 모드 구현 (펌웨어 손상 시 안전 모드 부팅)
- [ ] 관련 규격 인증 확인 (CE, FCC, KC 등 필요 시)

## 4. 요약 — 장기 운용 안정성 3대 원칙

**첫째, 결정론적으로 설계하라**
동적 할당을 최소화하고, 모든 리소스 사용량이 컴파일 타임에 예측 가능해야 한다. 런타임에 "아마 될 것이다"는 임베디드에서 통하지 않는다.

**둘째, 실패를 전제하고 설계하라**
센서는 고장나고, 통신은 끊기고, 전원은 불안정하다. 모든 외부 의존성에 타임아웃, 재시도, 폴백(fallback) 로직을 넣어라. 워치독은 선택이 아니라 필수다.

**셋째, 측정하지 않으면 모르는 것이다**
스택 사용량, 힙 여유 공간, 통신 에러율, 동작 온도 — 추측하지 말고 실측하라. 특히 장기 운용에서만 나타나는 문제(메모리 누수, 힙 프래그멘테이션, 타이머 오버플로우)는 단기 테스트로는 발견할 수 없다.
