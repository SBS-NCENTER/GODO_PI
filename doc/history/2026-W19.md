### issue#19 — EDT 2D Felzenszwalb 3-way 병렬화

issue#11에서 ship된 `ParallelEvalPool` primitive를 재사용해서 `build_likelihood_field` 함수 (Felzenszwalb 2D EDT)를 워커 0/1/2 cores로 분배. 핵심 API 확장: `parallel_for_with_scratch<Scratch>` 템플릿이 caller-owned `std::vector<S>` per_worker scratch와 caller-passed `deadline_ns_override`를 받음. 운영자가 D1~D10 lock한 결정 사항을 plan + Mode-A fold가 충실히 반영.

핵심 구조:

- **API 확장**: `ParallelEvalPool::parallel_for_with_scratch<Scratch>(begin, end, per_worker, fn, deadline_ns_override)` 템플릿. 캐스트는 `void*` shim으로 type-erased — pool은 ScratchT-agnostic 유지. 운영자 lock D1: runtime size guard는 **반드시 `if (per_worker.size() != worker_count()) { fprintf + std::abort(); }`**, NDEBUG-conditional `assert(...)` 매크로 금지 (release에서도 trip 강제). 새 build-grep `[edt-scratch-asserted]`가 `parallel_eval_pool.cpp` 함수 정의에서 이 패턴을 강제.
- **`build_likelihood_field` 시그니처**: `(grid, σ, ParallelEvalPool* pool = nullptr)`. nullptr ⇒ 기존 sequential 경로 byte-identical 보존. pool 있으면 column pass + row pass 각각 `parallel_for_with_scratch` 한 번씩 — 두 dispatch 사이 자연스러운 fork-join barrier.
- **Cache-aligned column partition** (D4): 16-float (64 B Cortex-A76 cache line) 단위로 round-down, 마지막 워커가 residue 가져감. Row pass는 row-block contiguous라 자연스럽게 정렬.
- **새 Tier-1 상수** `EDT_PARALLEL_DEADLINE_BASE_NS = 50 ms`, `EDT_PARALLEL_ANCHOR_DIM = 1000`. Formula `scale = max(1, max(W,H) / 1000)` — 1000×1000 → 50 ms/pass, 2000×2000 (`EDT_MAX_CELLS` edge) → 100 ms/pass. 운영자 lock m2: anchor 1000 ship + bench가 2000×2000 worker p99 > 80 ms 측정 시 anchor 750 fold (별도 작은 PR).
- **Sequential fallback graceful**: pool degraded ⇒ `per_worker[0]` scratch로 sequential 재실행 + 이후 `pool->degraded()` sticky flag로 short-circuit. `test_likelihood_field_parallel` case 5 / case 6이 이 경로 핀.
- **Bit-equality 보존**: 워커가 disjoint output subrange만 write (column pass: x range / row pass: y range), Gaussian 변환은 sequential. 16×16 + 256×256 + 1000×1000 grid 모두 FNV-1a memcmp byte-equal 핀.

D5 (no new TOML key) 보존 — `amcl.parallel_eval_workers = 1` (issue#11 lever)이 eval-pool과 EDT 둘 다 sequential 롤백 단일 lever. cold_writer M1 invariant 보존 — `<mutex>` 등 source text에 등장 안 함 (`[m1-no-mutex]` build-grep 그대로 clean). issue#11 `parallel_for(begin, end, fn)` 시그니처 byte-identical (test_parallel_eval_pool case (e) regression 핀).

### 배포 직후 critical defect — plan §3.7 / §4 self-inconsistency

운영자가 PR #99을 news-pi01에 배포 (`sudo bash production/RPi5/systemd/install.sh` + `systemctl restart godo-tracker`) 후 1분 49초 만에 journalctl에 떴음:

```
15:43:00  godo_tracker_rt: ParallelEvalPool ready (workers=3).
15:44:49  [pool-degraded] ParallelEvalPool: parallel_for join exceeded
          50 ms hard deadline (workers=3, range=[0,5000));
          falling back to inline-sequential mode for the remainder of
          this tracker process.
```

`range=[0,5000)`이 결정적 단서 — OneShot first-tick (또는 Live 재진입 시 N=5000 재시드) 시점의 dispatch가 첫번째로 50 ms hard deadline을 못 맞췄음. Pool은 영구 inline-sequential mode로 전환, 이후 모든 step은 sequential.

원인: Plan §3.7 first-tick anatomy는 N=5000을 parallel ~190 ms로 예측, §4는 flat 50 ms hard timeout. **190 ms 예상 작업을 50 ms로 자르면 첫-tick은 100% timeout 보장.** Mode-A round 1 reviewer가 §3.7 추가하면서 M2 닫았는데, Mode-A round 2 reviewer는 §3.7의 추가만 검증하고 §4 deadline과 cross-check 안 함. Mode-B reviewer도 §4 코드를 §4 spec과 비교했지만 §3.7과는 비교 안 함. **3개 review pass 모두 놓침**, 운영자 HIL이 즉시 캐치.

해결: Range-proportional deadline. `kJoinTimeoutBaseNs × max(1, range / kJoinTimeoutAnchorN)`, base 50 ms anchored on N=500. Steady-state Live N=500 → 50 ms 보존 (R5 worker-stall 가드 그대로); first-tick N=5000 → 500 ms (~2.5× margin over §3.7 ~190 ms parallel projection). Plan §3.5 / §4 / §6.1 / §7 R5 + post-deploy fold + production/RPi5/CODEBASE.md `(s)` invariant + W19 archive entries 모두 정합성 update. 한 commit (`bfbf671`)에 코드 + 메시지 + 문서 반영.

이번 세션에서 가장 중요한 lesson: "한 review pass가 §X를 검증하고 한 review pass가 §Y를 검증해도, **§X와 §Y의 상호작용이 self-inconsistency**일 수 있다." 운영자 HIL이 catch했지만, 다음 review에는 cross-section consistency check를 명시적으로 포함시켜야 함. Parent territory에서 feedback 메모리로 잠금 예정 (`feedback_cross_section_consistency_after_round_2_adds.md`).

### Range-proportional pattern — 재사용 가능한 레시피

Hot path가 workload size 10× 차이 (예: steady N=500 vs first-tick N=5000)를 한 architectural surface에서 dispatch할 때, flat hard timeout은 두 mode에 모두 적합 불가. `deadline ∝ max(1, workload / steady_anchor)` 패턴이 worker-stall guard도 보존하면서 heavy mode도 흡수. 적용 가능 후보:

- EDT row/col 패스 (W × H 그리드, map 크기에 비례; issue#19에서 다시 등장 가능)
- Map activate phased reload (re-prime 시간이 active map 크기에 비례)
- Phase 5 UE characterization runs (시나리오 길이에 따라 budget 배수 조정 가능)

Parent territory에서 project 메모리로 잠금 예정 (`project_range_proportional_deadline_pattern.md`).

### Live mode CPU thrashing — 정상 패턴 재분류 (구조적 발견 #1)

스물 여섯 번째 세션은 macroscopic 패턴(TID 3438307 cold-path AMCL 단독으로 40-60% CPU 지속, RT thread는 0.25%로 안전)만 잡고 끝남. 이번 세션에서는 작정하고 길게 봄:

운영자가 본 "all 4 cores 100%" 환영의 메커니즘: 1초 sampling 데이터로 60초 시야 100% 확률로 cpu 0/1/2 모두 visit, 30초 시야 80%, 10초 시야 33%. htop의 sub-second 막대 갱신 + cpu3의 짧은 깜빡임(kernel IRQ 처리)이 시각적으로 합쳐져 "4 코어 모두 100%"로 보였음. 메모리 파일 `.claude/memory/project_live_mode_cpu_thrashing_2026-05-05.md`에 "Resume + reclassification" 섹션 추가, 운영자 결정으로 "정상-as-designed" 잠금. 이 패턴은 issue#11 (Live pipelined-parallel multi-thread)의 정확한 motivator — 별건 버그가 아님.

### Process discovery — UDS hang after RPLIDAR I/O fallback to Idle (issue#X 후보)

세션 중간 production tracker (PID 1708747, ~12h uptime)가 갑자기 webctl `/api/health`에서 `tracker_unreachable` 응답. SPA에서도 "godo-tracker가 응답하지 않습니다" 메시지. 직접 UDS connect → `Connection refused`. 진단:

- 트래커 process는 살아있음 (모든 thread state=S)
- `ss -lx` → `LISTEN 0 0 /run/godo/ctl.sock.1708747.tmp` — backlog=0이 비정상 (kernel이 accept 거부)
- `journalctl` 마지막 라인 (09:22:05): `cold_writer: scan_frames(1) threw in Live: rplidar: grabScanDataHq failed 5 times in a row — returning to Idle.`

세션의 즉시 복구는 `systemctl restart godo-tracker`이 자연스럽게 해결 (Phase-0 HIL 직전이었음). **issue#X 후보** (Medium priority): `cold_writer.cpp::run_cold_writer` exception path가 UDS 서버 thread의 accept loop와 어떻게 상호작용하는지 조사. 정상 운영 환경에서 cpu 100% 부하 거의 없으니 재현 빈도 낮지만, 영구 진단성 향상이 있는 만큼 고치는 가치는 있음.

Round 2 driver fix는 `rplidar_node.cpp` source를 sed로 직접 patch (`M_PI - angle` → `-angle`). single-frame 방향성은 정상화 (`test_180check_after_fix`로 확인 — 정면 1m 벽이 PGM의 +x_world). 그러나 운영자가 cumulative mapping을 시도해 본 결과: 같은 벽을 여러 각도에서 보면 잔상처럼 새로 그려지고 누적되지 않음. 운영자 표현: "지도가 잔상처럼 남아있어 … 움직이면 계속 새로 그리는 듯해."

원인 진단: sed가 published `/scan`의 `angle_min/angle_max`를 conventional `[-π, π]` 범위에서 비표준 `[-2π, 0]` 범위로 시프트. ROS LaserScan을 소비하는 다운스트림 노드들 (rf2o + slam_toolbox) 중 일부가 `[-π, π]` 범위를 가정하는 internal logic을 갖고 있어서 scan-to-scan registration이 silently 잘못된 결과 산출 → slam_toolbox가 잘못된 상대 pose에 scan plot → 잔상.

해결: sed를 revert + upstream driver의 기존 `flip_x_axis: True` 런타임 파라미터로 동등 변환을 launch에서 적용. 이 파라미터는 `apply_index += scan_midpoint (= n/2)` shift를 publish_scan 데이터 채우기 루프에서 트리거. 배열 인덱스 공간에서의 n/2 shift = angle 공간에서의 180° 회전. 이게 upstream `M_PI - angle` (published angle 값 공간에서의 180° 회전)과 정확히 합쳐져서 published-vs-physical 대응 관계는 identity (정확한 orientation), AND 양쪽 piece가 독립적으로 conventional `[-π, π]` 범위를 보존.

운영자 HIL final 검증 (`test_180check_dockercheck_v1`): single-frame 정상 + cumulative 정상 + 30° yaw + ~70° (≈90°) yaw 모두 정상. "이대로 머지하면 될 것 같아."

메모리: `project_rplidar_cw_vs_ros_ccw.md` SSOT 동일 entry 업데이트 (chosen fix + why-not-source-patch).

### 핵심 구조 발견 #4 — 운영자 yaw 방향 직관은 PR #81이 아니라 수학 표준 컨벤션
